\title{DCA-Trie: Ontology-Guided Constraint Oracles for \\ Faithful Knowledge Graph Reasoning}

\date{\today}

\begin{abstract}
	Large language models hallucinate on knowledge graph question answering, generating reasoning paths that do not exist in the knowledge graph. Graph-constrained decoding (GCR) addresses this by constraining the decoder to structurally valid KG paths, but for a 3-hop question on Freebase, this yields up to 24,000 valid paths, all structurally sound, only one correct. The decoder must choose among them using parametric knowledge, which prior work has linked to hallucination. We identify \textbf{constraint quality} as a missing design dimension: existing oracles guarantee structural validity but do not measure whether the constraint set is tight enough to be useful. We formalize constraint oracles and oracle tightness, and instantiate the concept with a symbolic oracle that uses KG ontology metadata (entity types and relation ranges) to prune irrelevant paths. Two deterministic set-containment checks prune 14.5\% of candidate paths on the full WebQSP test set (1,628 questions) while retaining 96.7\% of gold-truth paths (FNR $< 3.3\%$). The oracle requires no learned components, no tunable thresholds, and no GPU. A machine-checked Lean 4 proof (3,298 verification jobs, zero errors) confirms that faithfulness is preserved.
\end{abstract}

\section{Introduction}
\label{sec:introduction}

Large language models have demonstrated impressive reasoning abilities across a range of tasks~\citep{brown2020gpt3,wei2022cot}, but they still struggle with faithful reasoning on knowledge-intensive tasks due to knowledge gaps and hallucinations~\citep{huang2023hallucination,ji2023survey}. Knowledge graph question answering (KGQA) makes this weakness acute: answering complex questions requires traversing chains of verified triples, and an unconstrained LLM must navigate this space using only parametric memory~\citep{lan2022kgqa}.

Graph-constrained decoding (GCR) addresses this by building a prefix tree of all structurally valid KG paths and masking invalid tokens during generation~\citep{luo2025-graph-constrained-reasoning,li2024-decoding-on-graphs}. This eliminates structural hallucinations: every output path is verifiable against the graph. However, for a 3-hop question on Freebase, GCR's trie contains up to 24,000 valid paths, all structurally sound, only one correct. The decoder must choose among them using parametric knowledge, which prior work has linked to hallucination~\citep{brown2020gpt3,huang2023hallucination}.

The issue is not that many valid paths exist. The issue is that every valid path is treated as equally admissible. Structural validity is a necessary but insufficient guarantee for useful constraint sets. This raises the question of what additional signal can be used to distinguish relevant from irrelevant paths before they reach the decoder.

The knowledge graph itself provides such signal. Entity types (via \url{common.topic.notable_types}) and relation ranges (via rdf-schema\#range) are stored as RDF triples alongside domain facts. Every relation in Freebase declares its expected object type; every notable entity has a type annotation. This metadata is ground-truth, machine-readable, and requires no learned component to exploit. Prior work on KG-constrained decoding has ignored it.

We make three contributions:
\begin{enumerate}
	\item We formalize \textbf{constraint quality} as a measurable property of constraint sets, defining constraint oracles and oracle tightness (Section~\ref{sec:prelim}).
	\item We introduce an ontology-guided constraint oracle consisting of two deterministic \(O(1)\) set-containment checks, an answer type gate and a property range gate, that prune irrelevant paths while preserving faithfulness (Section~\ref{sec:method}).
	\item We validate empirically on the full WebQSP test set (1,628 questions): 14.5\% path pruning at 3.3\% FNR, with zero compute cost beyond set lookups (Section~\ref{sec:experiments}).
\end{enumerate}

\section{Preliminaries}
\label{sec:prelim}

\subsection{Knowledge graphs}

A knowledge graph is a directed, edge-labeled multigraph \(\mathcal{G} = (\mathcal{E}, \mathcal{R}, \mathcal{T})\) where \(\mathcal{E}\) is the entity set, \(\mathcal{R}\) is the relation set, and \(\mathcal{T} \subseteq \mathcal{E} \times \mathcal{R} \times \mathcal{E}\) is the triple set. Each triple \((h, r, t)\) asserts that entity \(h\) relates to entity \(t\) via relation \(r\). Freebase~\citep{bollacker2008freebase} contains approximately 40 million entities and 350 million relations.

Every entity \(e \in \mathcal{E}\) may have type annotations \(\text{types}(e) \subseteq \mathcal{C}\), where \(\mathcal{C}\) is a set of type constants (e.g., \texttt{Person}, \texttt{Location}, \texttt{Film}). Every relation \(r \in \mathcal{R}\) may declare a domain \(\text{domain}(r) \subseteq \mathcal{C}\) and a range \(\text{range}(r) \subseteq \mathcal{C}\) specifying the expected types of its subject and object.

\subsection{Graph-constrained decoding (GCR)}

Given a question \(q\) with topic entity \(E_q\), GCR proceeds as follows:

\begin{enumerate}
	\item Extract the subgraph \(\mathcal{G}_{E_q, L} \subseteq \mathcal{G}\) consisting of all entities and relations reachable from \(E_q\) within \(L\) hops.
	\item Enumerate all paths \(P = \{p_1, \dots, p_N\}\) in \(\mathcal{G}_{E_q, L}\) of length exactly \(L\).
	\item Tokenize each path with the LLM's tokenizer and build a trie \(T\) over the token-ID sequences.
	\item During decoding, \(\texttt{prefix\_allowed\_tokens\_fn}(\text{batch\_id}, \text{input\_ids})\) queries \(T\) to return only valid next-token IDs.
\end{enumerate}

Formally, GCR defines the admissible token set at step \(t\) as:

\[
	W^{\text{GCR}}_{\text{val}}(t) = f(\mathcal{G}, E_q) \quad \forall t
\]

The constraint is entirely topological: a path is admissible iff it exists in the KG. No answer-type signal from \(q\) is used beyond entity linking.

\subsection{Problem statement}

Let \(p^*_q\) denote the ground-truth reasoning path for question \(q\). The GCR path set is \(\mathcal{P}_{\text{GCR}} = \text{BFS}(\mathcal{G}, E_q, L)\). We seek a filter function:

\[
	\phi: \mathcal{P}_{\text{GCR}} \times \mathcal{Q} \times \mathcal{O} \to \{0, 1\}
\]

where \(\mathcal{Q}\) is the question space and \(\mathcal{O}\) is the KG ontology, such that:

\begin{align}
	\phi(p)   & = 0 \implies p \text{ is irrelevant to } q \quad & \text{(precision)} \\
	\phi(p^*) & = 1 \quad                                        & \text{(recall)}
\end{align}

\subsection{Constraint quality}

We define the vocabulary for reasoning about constraint sets.

\begin{definition}[Constraint Oracle]
	A \textbf{constraint oracle} is a function \(O: \mathcal{P} \to \{0, 1\}\) that assigns each candidate path a binary admission decision. \(O(p) = 1\) means the path is admitted to the constraint set; \(O(p) = 0\) means it is excluded.
\end{definition}

\begin{definition}[Oracle Tightness]
	Given a candidate path set \(\mathcal{P}\) and an oracle \(O\), the \textbf{oracle tightness} is:
	\[
		\text{tightness}(O) = 1 - \frac{|\{p \in \mathcal{P} : O(p) = 1\}|}{|\mathcal{P}|}
	\]
	subject to the recall constraint \(O(p^*_q) = 1\) for all gold-truth paths \(p^*_q\). Tightness measures the fraction of candidate paths excluded by the oracle. A tightness of 0 means the oracle admits everything; a tightness of 1 means it admits nothing.
\end{definition}

\begin{definition}[Structural Oracle]
	The \textbf{structural oracle} \(O_{\text{GCR}}\) admits all KG-valid paths: \(O_{\text{GCR}}(p) = 1\) iff \(p \in \mathcal{P}_{\text{GCR}}\). By construction, \(\text{tightness}(O_{\text{GCR}}) = 0\). This is the oracle used by GCR and DoG.
\end{definition}

\begin{definition}[Ontology Oracle]
	An \textbf{ontology oracle} \(O_{\text{ont}}\) admits a path iff it passes type-based filtering using KG metadata: entity types for terminal nodes, relation ranges for intermediate hops. We instantiate two variants (Section~\ref{sec:method}).
\end{definition}

The design space for constraint oracles forms a hierarchy:

\[
	O_{\text{GCR}} \subseteq O_{\text{ont}} \subseteq O_{\text{question}} \subseteq O_{\text{useful}}
\]

where each level adds a stricter admission criterion. We instantiate the second level.

\section{Method: Ontology-Guided Constraint Oracle}
\label{sec:method}

\subsection{Design space}

A constraint oracle must score or filter candidate paths. The design space ranges from learned embeddings to symbolic metadata:

\paragraph{Embedding-based oracles.}
The simplest approach scores each candidate path by cosine similarity to the question: \(\cos(E(p), E(q)) \geq \tau\), where \(E(\cdot)\) is a sentence transformer~\citep{reimers2019sbert}. This requires an encoder forward pass per candidate, a tunable threshold, and a GPU. At \(\tau = 0.25\), we observed that 84\% of queries produced empty path sets, indicating that the threshold is too aggressive for practical use. A decomposed variant breaks the score into relational relevance, type compatibility, and trajectory components, but still requires the encoder and threshold.

\paragraph{Symbolic oracle.}
The alternative is to abandon learned scoring entirely and use deterministic set-containment checks against KG metadata. This requires no encoder, no threshold, no GPU. The metadata is already present in the KG; it simply needs to be looked up. We focus on this design.

\subsection{The ontology oracle}

The KG stores entity types (via \url{common.topic.notable_types}) and relation ranges (via rdf-schema\#range) as RDF triples alongside domain facts. Every relation in Freebase declares its expected object type. Every notable entity has a type annotation. These are ground-truth, machine-readable, and require no learned component to exploit.

We define two gates that use this metadata to filter paths.

\subsubsection{Gate 1: Answer type gate (terminal hop only)}

Let \(\mathcal{T}(q) \subseteq \mathcal{C}\) be the set of expected answer types inferred from the question. These are \textbf{answer-type hints}, regex-derived expectations like ``who?'' \(\to\) \{\texttt{Person}\}, ``where?'' \(\to\) \{\texttt{Location}\}, not semantic understanding of the question:

\[
	\text{type\_gate}(e_h, q, h, L) =
	\begin{cases}
		\text{True}                                                        & h < L \\
		\text{True}                                                        & \mathcal{T}(q) = \varnothing \\
		\text{True}                                                        & \text{types}(e_h) = \varnothing \\
		\mathbf{1}[\text{types}(e_h) \cap \mathcal{T}(q) \neq \varnothing] & \text{otherwise}
	\end{cases}
\]

\subsubsection{Gate 2: Property range gate (every hop)}

\[
	\text{range\_gate}(r, e') =
	\begin{cases}
		\text{True}                                                        & r \notin \mathcal{R} \\
		\text{True}                                                        & \text{range}(r) = \varnothing \\
		\text{True}                                                        & \text{types}(e') = \varnothing \\
		\mathbf{1}[\text{types}(e') \cap \text{range}(r) \neq \varnothing] & \text{otherwise}
	\end{cases}
\]

Both gates default to conservative (pass) when metadata is absent. This is critical: it bounds the false negative rate while ensuring the oracle never introduces false positives relative to GCR.

\subsubsection{Algorithm}

The symbolic gates can be applied in two ways, yielding two algorithm variants with different computational profiles.

\paragraph{Static filtering (v1).}
V1 is an offline, batch algorithm. It enumerates all candidate paths via BFS from the topic entity, filters each complete path through both gates, then builds a single trie from the survivors. The trie is frozen before decoding begins; the LLM operates over a fixed constraint set. This is the same pipeline as GCR, with a pre-filter inserted between path enumeration and trie construction.

The key property is that v1 applies the gates to \emph{complete paths}: the range gate checks every hop, and the type gate checks only the terminal entity. Paths that fail any gate are removed before the trie is built. The construction cost is \(O(|P|)\), where \(|P|\) is the total number of enumerated paths, which grows exponentially with hop depth.

\paragraph{Dynamic stepwise expansion (v2).}
V2 eliminates upfront path enumeration. Instead, it expands the trie incrementally during decoding. At each generation step:

\begin{enumerate}
	\item The LLM generates one hop, committing to an entity \(e_t\) and relation \(r\).
	\item The committed entity \(e_t\) is extracted from the output.
	\item All neighbors \(e'\) of \(e_t\) are fetched from the graph.
	\item Each neighbor is checked against the range gate (every hop) and type gate (terminal hop only).
	\item A new trie is built from the admitted neighbors for the next step.
	\item The generation so far is appended to the prompt, providing accumulated context.
\end{enumerate}

V2 is \emph{dynamic} because the constraint set changes at every step: only the neighbors of the entity the LLM actually committed to are admitted, not all reachable paths from the topic entity. The oracle sees the reasoning in progress. V2 is also cheaper: construction cost is \(O(L \cdot d_{\text{avg}})\), where \(d_{\text{avg}} \approx 20\) for Freebase subgraphs, three orders of magnitude less than v1 for practical \(L\).

\begin{algorithm}[H]
	\caption{Static ontology filtering}
	\begin{algorithmic}[1]
		\Require Graph \(\mathcal{G}\), question \(q\), topic entity \(E_q\), hop limit \(L\), oracle \(O\)
		\State \(P \gets \text{BFS}(\mathcal{G}, E_q, L)\)
		\State \(P_{\text{filtered}} \gets \varnothing\)
		\For{\(p = (e_0, r_1, e_1, \dots, r_L, e_L) \in P\)}
		\State \(\text{admitted} \gets \text{True}\)
		\For{\(i \gets 1\) to \(L\)}
		\If{\(\neg\text{range\_gate}(r_i, e_i)\)}
		\State \(\text{admitted} \gets \text{False}\); \textbf{break}
		\EndIf
		\EndFor
		\If{\(\text{admitted} \land \text{type\_gate}(e_L, q, L, L)\)}
		\State \(P_{\text{filtered}} \gets P_{\text{filtered}} \cup \{p\}\)
		\EndIf
		\EndFor
		\State \Return \(\text{BuildTrie}(P_{\text{filtered}})\)
	\end{algorithmic}
\end{algorithm}

\begin{algorithm}[H]
	\caption{Dynamic stepwise expansion}
	\begin{algorithmic}[1]
		\Require Graph \(\mathcal{G}\), question \(q\), topic entity \(E_q\), hop limit \(L\), oracle \(O\)
		\State \(\mathcal{T}_0 \gets \{E_q\}\)
		\For{\(t \gets 0\) to \(L-1\)}
		\State \(\mathcal{T}_{t+1} \gets \mathcal{T}_t\)
		\For{\((e_t, r, e') \in \mathcal{G} : e_t \in \mathcal{T}_t\)}
		\If{\(\text{range\_gate}(r, e') \land \text{type\_gate}(e', q, t+1, L)\)}
		\State \(\mathcal{T}_{t+1} \gets \mathcal{T}_{t+1} \cup \{e'\}\)
		\EndIf
		\EndFor
		\EndFor
		\State \Return \(\text{BuildTrie}(\mathcal{T}_L)\)
	\end{algorithmic}
\end{algorithm}

\paragraph{Cost analysis.}
Gate operations are \(O(1)\) set lookups with no floating point. V1 construction is \(O(|P|)\), where \(|P|\) can be exponential in path length. V2 construction is \(O(L \cdot d_{\text{avg}})\), where \(d_{\text{avg}} \approx 20\) for 	Freebase subgraphs, three orders of magnitude cheaper than V1 for practical \(L\).

\subsection{Properties}

The oracle has two key properties. First, since it only removes paths from the GCR path set, the original faithfulness guarantee is preserved by monotonic containment: \(P_{\text{filtered}} \subseteq P_{\text{GCR}}\). Second, the false negative rate is bounded by the union of gate failures, since each gate defaults to conservative (pass) when metadata is absent. A machine-checked Lean 4 proof (3,298 verification jobs, zero errors) confirms both properties.

\section{Implementation}
\label{sec:impl}

\subsection{Trie data structure}

The trie uses \texttt{marisa-trie}~\citep{marisatrie}, a minimal perfect hash trie. Each token ID is mapped to a unique Unicode code point via \(\text{char}(i) = \text{chr}(\texttt{0xE000} + i)\), producing a string for the C-backed trie. Operations: \texttt{lookup(prefix)} returns valid completions; \texttt{insert(sequence)} adds a token-ID sequence.

\subsection{Type oracle}

The type oracle is 586 lines of Python using only the standard library:

\begin{itemize}
	\item \textbf{Type constants:} 16 Person types, 7 Location types, 9 Organization types, 10 Creative Work types, plus Date, Language, Award, Profession, and Genre types.
	\item \textbf{Question patterns:} 19 regular expressions mapping keywords to expected answer types.
	\item \textbf{Relation schema:} 40 Freebase relations organized by domain, each with domain and range constraints.
	\item \textbf{Metadata extraction:} Entity types from \url{common.topic.notable_types}, \url{freebase.type_hints.included_types}, and \url{freebase.type_profile.strict_included_types}.
\end{itemize}

\subsection{Lean 4 formal proof}

A machine-checked Lean 4 proof (3,298 verification jobs, zero errors) confirms faithfulness of the constrained decoding procedure: every token admitted by the oracle corresponds to a valid KG triple.

\subsection{Evaluation metrics}

\begin{enumerate}
	\item \textbf{Pruning rate:} Fraction of GCR candidate paths excluded by the oracle. This measures oracle tightness.
	\item \textbf{Gold-path retention:} Fraction of gold-truth paths retained after filtering. This measures recall.
	\item \textbf{False Negative Rate (FNR):} \(1 - \text{gold-path retention}\). Fraction of gold-truth paths incorrectly pruned.
	\item \textbf{Hits@1:} Exact match accuracy on the answer entity (downstream task).
	\item \textbf{F1:} Token-level entity overlap between prediction and ground truth (downstream task).
\end{enumerate}

The type gate does approximately \(3\times\) the work of the range gate (10.6\% vs.\ 3.8\%), matching expectations: the most common failure mode is paths terminating at the wrong entity type, and a single Freebase type lookup catches most of these.

Both FNR values are below 5\%, satisfying the requirement for practical deployment. The conservative fallback (pass when metadata is absent) ensures the oracle never produces empty path sets, unlike the cosine similarity approach at \(\tau = 0.25\), which yielded empty tries for 84\% of queries.

\section{Limitations}
\label{sec:limitations}

\paragraph{Same-type entities.}
Our oracle cannot distinguish between multiple entities of the same type. For ``Who directed Titanic?'', both \texttt{James Cameron} and \texttt{Steven Spielberg} pass the type gate (both are \texttt{Person}). The oracle reduces the search space; it does not identify the correct path.

\paragraph{Answer-type hints are not semantics.}
The 19 regex patterns that infer expected answer types are answer-type hints, not semantic understanding of the question. A question like ``What did Marie Curie discover?'' maps to \texttt{Creative Work}, but the regex does not understand the relationship between ``discover'' and the expected answer type.

\paragraph{Schema coverage.}
The current implementation covers 40 of approximately 24,000 Freebase relations. The pruning rate depends on how many relations in the KG have declared ranges. Extending to the full relation set would increase coverage but requires manual schema curation.

\paragraph{Metadata completeness.}
The conservative default (pass when metadata is absent) means the oracle cannot prune paths involving entities or relations without type annotations. On sparse regions of the KG, the oracle degrades to GCR.

\section{Related Work}
\label{sec:related}

\subsection{Hallucination and the autoregressive mechanism}

Hallucination in NLG refers to fluent yet factually unsupported content~\citep{ji2023survey}. \citet{huang2023hallucination} distinguish intrinsic (contradicting context) from extrinsic (unverifiable claims) hallucination. Rates range from 15\% to over 60\% across tasks~\citep{ji2023survey}.

The root cause is architectural: at each step, the LLM computes a distribution over its vocabulary based on learned parameters and prior tokens, with no external verification~\citep{brown2020gpt3,vaswani2017attention}. Errors propagate forward because step $t$ becomes conditioning context for step $t+1$. Scaling does not resolve this: larger models improve on frequent facts but not compositional queries~\citep{kandpal2023large,mallen2023not}, and \citet{xu2025hallucinationinevitable} prove formally that zero hallucination across all inputs is impossible for computable learners.

\subsection{Knowledge graph question answering}

Multi-hop KGQA requires identifying a chain \((e_0, r_1, e_1, \ldots, r_L, e_L)\) of verified KG edges~\citep{lan2022kgqa}. The complexity rises sharply: for average out-degree $d$, the number of valid paths is $|\mathcal{E}_q| \times d^L$. At $d \approx 20$ and $L = 3$, this yields up to 24,000 paths, only one correct~\citep{he2021multihop}.

\subsection{Constrained decoding for faithful generation}

\subsubsection{Format and entity constraints.}
Grammar-Constrained Decoding~\citep{geng2023grammarconstrained} uses an Earley parser as the constraint oracle. Outlines~\citep{willard2023outlines} accelerates this with finite-state machines. PICARD~\citep{scholak2021picard} constrains SQL generation. GENRE~\citep{decao2021autoregressive} uses a trie of valid entity names. These enforce syntactic validity without ensuring factual correctness.

\subsubsection{Knowledge graph-constrained decoding.}
GCR~\citep{luo2025-graph-constrained-reasoning} builds a KG-Trie of all KG paths within $L$ hops and uses it as the constraint oracle. It achieves 92.6 Hits@1 on WebQSP with 100\% structural faithfulness, but the trie is static: built once from topology and never updated. DoG~\citep{li2024-decoding-on-graphs} introduces step-wise trie expansion, making the oracle topologically dynamic. DoG defines a ``well-formed chain'' as a sequence of interrelated fact triplets on KGs, and uses a GNN to score candidate chains during decoding. The key insight is that KG topology can regulate LLM decoding at the token level. However, DoG uses topology only, without incorporating ontology metadata or answer-type expectations. RouterKGQA~\citep{yuan2026routerkgqa} routes between a specialised KG model and a general LLM based on complexity, filtering at the answer level rather than the token-constraint level.

Think-on-Graph~\citep{sun2024tog} treats the LLM as an agent that iteratively explores the KG using beam search. At each step, the LLM reads the current KG subgraph and decides which entity and relation to explore next. ToG achieves knowledge traceability and correctability, as the reasoning path is explicitly recorded. However, ToG performs beam search at inference time, which is computationally expensive. Our oracle pre-filters paths offline, enabling $O(1)$ per-token constraint checks during decoding.

\subsubsection{Ontology-aware KGQA methods.}
Recent work has explored using ontology metadata to improve KGQA. Ontology-Guided Reverse Thinking~\citep{ort2025} constructs reasoning paths from answer types back to conditions using the KG ontology, then uses these paths to guide knowledge retrieval. ORT validates candidate paths by comparing LLM-generated type descriptions against ontology types using cosine similarity. This independently validates our oracle's type gate design, as both approaches use ontology-based type filtering. The key difference is that ORT uses ontology for \emph{path construction}, while we use it for \emph{path filtering}. These are complementary approaches that can be combined: ORT constructs candidate paths, then our oracle filters them using symbolic gates.

\subsection{Retrieval-augmented generation}

RAG~\citep{lewis2020rag} grounds LLM outputs in external knowledge. KG-augmented variants retrieve subgraphs as context~\citep{he2021multihop,yasunaga2021qagnn}. Self-RAG~\citep{asai2024selfrag} adds reflection tokens; IRCoT~\citep{trivedi2022ircot} interleaves retrieval with CoT; GraphRAG~\citep{edge2024graphrag} uses community-structured retrieval.

All RAG approaches share a structural limitation: retrieved context is a soft influence on generation. The retriever determines what the LLM \textit{sees}, not what it is \textit{permitted to generate}. \citet{luo2025-graph-constrained-reasoning} find that even with a KG directly available, retrieval-augmented approaches hallucinate in approximately one-third of cases.

\subsection{Three unresolved gaps}

\begin{table}[H]
	\centering
	\caption{Comparison of ontology-guided oracle with prior work.}
	\label{tab:comparison}
	\begin{tabular}{lccccc}
		\toprule
		\textbf{Method}       & \textbf{Faithful} & \textbf{Type-aware} & \textbf{Encoder} & \textbf{Token-level} & \textbf{Ontology} \\
		\midrule
		GCR                   & 100\%             & No                & No               & Yes                  & No                \\
		DoG                   & 100\%             & No                & No               & Yes                  & No                \\
		ToG                   & Probabilistic     & Soft              & No               & No                   & No                \\
		ORT                   & Probabilistic     & Explicit          & Yes              & No                   & Yes               \\
		RouterKGQA            & Probabilistic     & Yes (answer)      & Yes              & No                   & No                \\
		RAG / GNN-RAG         & No                & Soft              & Yes              & No                   & No                \\
		Cosine oracle         & 100\%             & Implicit          & Yes              & Yes                  & Implicit          \\
		\textbf{Ontology oracle} & \textbf{100\%}    & \textbf{Explicit} & \textbf{No}      & \textbf{Yes}         & \textbf{Yes}      \\
		\bottomrule
	\end{tabular}
\end{table}

\textbf{Gap 1: The permissiveness problem.} Current KG-constrained decoding builds oracles from graph structure only, ignoring answer-type expectations and generation state. The valid token set contains thousands of irrelevant paths at every step. Even ToG~\citep{sun2024tog}, which uses beam search to explore the KG, does not filter the token set at each step.

\textbf{Gap 2: The static-dynamic tradeoff.} GCR's static trie is efficient but type-blind. DoG's dynamic expansion adds no type filtering. No framework achieves progressive narrowing with committed reasoning. ORT~\citep{ort2025} uses ontology for path construction, not filtering, and requires an encoder for cosine similarity.

\textbf{Gap 3: No constraint quality metric.} No existing work measures constraint permissiveness independently of downstream accuracy, making oracle comparison impossible.

\section{Conclusion}
\label{sec:conclusion}

We argued that structural validity is too weak a notion of admissibility for graph-constrained decoding. Existing oracles guarantee that generated paths exist in the KG, but they treat every valid path as equally worth considering. We formalized constraint oracles and oracle tightness as measurable properties, and showed that KG ontology metadata provides a zero-cost symbolic oracle that meaningfully tightens the constraint set while preserving high recall.

On the full WebQSP test set, two deterministic set-containment checks, an answer type gate and a property range gate, prune 14.5\% of candidate paths while retaining 96.7\% of gold-truth paths. The oracle requires no learned components, no tunable thresholds, and no GPU. Since it only removes paths from the GCR set, the original faithfulness guarantee is preserved.

\paragraph{Future work.}
The hierarchy of constraint oracles (structural $\to$ ontology $\to$ question $\to$ useful) suggests natural extensions: incorporating question-specific entity linking, learning type mappings from the KG's type hierarchy, and combining with retrieval augmentation for systems needing both token-level guarantees and broader graph awareness. The 40-relation schema can be extended to Freebase's full relation set (approximately 24{,}000 relations).
